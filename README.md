# ATM Session-Based REST API

**C# · ASP.NET Core · MSDI · SOLID · GRASP · Hexagonal Architecture**

Сервис банкомата с сессионной авторизацией, разделением на слои и тестированием бизнес-логики через моки репозиториев. Проект сделан как учебная лабораторная, но без архитектурного цирка: логика отделена от веб-слоя, зависимости инвертированы, хранение данных абстрагировано, а операции со счётом всегда фиксируются в истории.

## О проекте

Это backend-приложение для работы с банковскими счетами через HTTP API. Пользователь создаёт сессию, получает ключ и дальше выполняет операции уже в контексте этой сессии. В системе есть два типа сессий:

- **Пользовательская**: создаётся по номеру счёта и PIN-коду.
- **Админская**: создаётся по системному паролю.

Все операции со счётом сохраняют историю, чтобы потом не гадать, куда исчезли деньги. Люди, конечно, и так стараются не помнить, но лог лучше.

## Функциональность

- создание сессии пользователя или администратора
- создание счёта
- просмотр баланса
- пополнение счёта
- снятие денег со счёта
- просмотр истории операций
- авторизация операций по ключу сессии

## Архитектура

Проект построен в духе **hexagonal architecture** с возможной близостью к **clean architecture / onion architecture**:

- **Presentation layer** — ASP.NET Core Web API на контроллерах
- **Application layer** — сценарии использования и координация бизнес-операций
- **Domain layer** — сущности, правила и инварианты предметной области
- **Infrastructure layer** — in-memory реализации репозиториев и прочие технические детали

### Принципы

- **SOLID** — зависимости направлены через абстракции, а не через конкретные классы
- **GRASP** — ответственность распределена по слоям и сущностям без лишней магии
- **Dependency Injection** — связывание абстракций и реализаций через `Microsoft.Extensions.DependencyInjection`
- **Модульная регистрация** — каждый модуль описывает собственные extension methods для регистрации зависимостей
- **In-memory storage** — реализация хранения данных не протекает в бизнес-логику

## Что важно по требованиям

- используется **ASP.NET Core Web API**
- используются **контроллеры**, без Minimal API
- все операции выполняются через **ключ сессии**
- сессии и их данные сохраняются через **репозитории**
- системный пароль вынесен в **конфигурацию**
- некорректные операции возвращают соответствующий HTTP-ответ, например `400 BadRequest` или `401 Unauthorized`
- тесты проверяют **бизнес-логику**, а не детали хранения данных или web-представления
- в тестах используются **моки репозиториев**

## Тестируемые сценарии

- **Снятие денег**
  - при достаточном балансе счёт сохраняется с обновлённым балансом
  - при недостаточном балансе сервис возвращает ошибку
- **Пополнение счёта**
  - счёт сохраняется с корректно обновлённым балансом

## Пример структуры решения

```text
src/
  Atm.Api/                 # ASP.NET Core Web API
  Atm.Application/         # use cases, DTO, сервисы
  Atm.Domain/              # сущности, value objects, интерфейсы
  Atm.Infrastructure/      # in-memory репозитории, конфигурация
  Atm.Tests/               # unit tests
```

Структура может отличаться, но смысл простой: UI не должен знать, как именно живут данные в памяти, а домен не должен зависеть от того, что внешний мир снова придумал новый фреймворк.

## Сущности предметной области

Обычно в проекте присутствуют:

- **Account** — банковский счёт
- **Session** — сессия пользователя или администратора
- **OperationHistoryItem** — запись об операции

## Поток работы

1. Клиент создаёт сессию.
2. Система возвращает ключ сессии (`Guid` или аналогичный идентификатор).
3. Клиент передаёт ключ сессии в последующих запросах.
4. Система проверяет авторизацию для операции.
5. Бизнес-логика выполняет действие и сохраняет изменения.
6. Любое изменение счёта добавляет запись в историю операций.

- **C# / .NET**
- **ASP.NET Core**
- **Microsoft.Extensions.DependencyInjection**
- **NSubstitute** для unit-тестов и моков

## Про качество кода

Проект ориентирован на следующие практики:

- разделение ответственности по слоям
- явные зависимости через интерфейсы
- тестирование бизнес-логики отдельно от инфраструктуры
- простые и предсказуемые сценарии использования
- отсутствие привязки к конкретному способу хранения данных

## Почему это выглядит как нормальный проект

Потому что в нём есть не только набор эндпоинтов, но и архитектура, которая переживёт больше одного учебного семестра. Доменная логика не размазана по контроллерам, репозитории можно подменять в тестах, а правила работы с аккаунтами не зависят от того, где физически лежат данные.

## Запуск

```bash
# восстановление зависимостей
dotnet restore

# сборка
dotnet build

# запуск API
dotnet run --project src/Atm.Api

# тесты
dotnet test
```

## Примечания

- Проект соответствует учебному заданию по многослойной архитектуре.
- Реализация хранения данных выполнена как in-memory, чтобы не смешивать бизнес-логику и инфраструктуру.
- Веб-часть намеренно не усложняется, потому что цель здесь — архитектура, а не доказательство религиозной преданности идеальному REST.
